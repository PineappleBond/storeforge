# 完成前验证

> 在标记任何任务完成前，运行全面的检查清单。`/sf:verify` 或 `storeforge-executing` 每个子任务完成后自动触发。

## 定位

本 skill 是任务完成前的最后一道防线。当 `storeforge-executing` 的某个子任务开发完成、测试通过、准备提交之前，自动调用此 skill 运行四项检查清单（代码规范、测试、Git、文档），确保交付质量。

**触发方式**：
- 用户手动调用 `/sf:verify`
- `storeforge-executing` 在每个子任务完成后自动调用

**依赖**：本 skill 依赖 `storeforge-harness` 提供 domain 反模式检查结果。

---

## 验证清单

### 一、代码规范检查

检查项来源：`storeforge-harness` 的约束引擎 + 通用代码规范。

| # | 检查项 | 说明 | 判定 |
|---|--------|------|------|
| 1.1 | Domain 反模式 | 是否违反当前 domain 的 BLOCK 级别反模式清单（见 `storeforge-harness`） | 通过/失败 |
| 1.2 | 命名规范 | 变量/函数/类型/文件命名是否符合对应 domain skill 的约定 | 通过/失败 |
| 1.3 | 错误处理 | 所有错误路径是否被正确处理（无裸 `error` 忽略、无 panic 传播） | 通过/失败 |
| 1.4 | 金额规范 | 金额是否使用 `int64`（分），舍入是否为 `ROUND_HALF_UP` | 通过/失败 |
| 1.5 | 状态机规范 | 订单状态转换是否经过合法路径（见 `order-state-machine.md`） | 通过/失败/N/A |
| 1.6 | 认证规范 | JWT 是否 RS256/ES256，Admin/User 角色分离 | 通过/失败/N/A |

> 如果当前任务涉及多个 domain（如后端 + 前端），加载所有相关 domain 的反模式清单合并检查。

### 二、测试检查

| # | 检查项 | 说明 | 判定 |
|---|--------|------|------|
| 2.1 | L1 单元测试通过 | 对应端的 L1 测试全部通过，race detector（Go）无报错 | 通过/失败 |
| 2.2 | 覆盖率达标 | 当前变更涉及的代码覆盖率满足 `storeforge-testing` 要求的阈值 | 通过/失败 |
| 2.3 | L2 场景覆盖（如适用） | 当前变更涉及的 L2 集成场景已被覆盖（见 `storeforge-testing` 13 个必测场景） | 通过/失败/N/A |
| 2.4 | 测试数据无硬编码 | 测试数据使用 UUID 生成的 ID，无硬编码时间/ID/Token | 通过/失败 |
| 2.5 | 测试反模式 | 无精确时间戳断言、无列表顺序依赖、无无重试异步断言 | 通过/失败 |

### 三、Git 检查

| # | 检查项 | 说明 | 判定 |
|---|--------|------|------|
| 3.1 | Commit message 格式 | 是否符合 Conventional Commits：`<type>(<scope>): <description>` | 通过/失败 |
| 3.2 | 变更文件完整性 | `git diff` 包含所有变更文件，无遗漏的已修改文件 | 通过/失败 |
| 3.3 | 无敏感信息 | 无 API Key、密码、Token、私钥等敏感信息被提交 | 通过/失败 |
| 3.4 | 单任务单提交 | 每个子任务对应一个独立 commit，无无关变更混入 | 通过/失败 |

### 四、文档检查

| # | 检查项 | 说明 | 判定 |
|---|--------|------|------|
| 4.1 | OpenAPI spec 更新 | 如有 API 变更（新增/修改/删除接口），`api/*.api` 文件是否已更新 | 通过/失败/N/A |
| 4.2 | 枚举注册 | 如有新增领域枚举，是否已注册到 `event/` 目录的 protobuf message | 通过/失败/N/A |
| 4.3 | 错误码注册 | 如有新增错误码，是否符合 `{MODULE}_{SEQ}` 命名空间且无冲突 | 通过/失败/N/A |

## 验证流程

### 逐项检查

按上述四大类逐一执行检查，每项检查结果为以下三种之一：

- **通过**：检查项满足要求
- **失败**：检查项不满足要求，需修复
- **N/A**：检查项不适用于当前任务变更（如未修改 API 则 4.1 为 N/A）

### 判定规则

```
所有检查项结果
  ↓
是否全部通过或 N/A？
  ├─ 是 → 验证通过，任务可标记为完成
  └─ 否 → 列出所有失败项 → 修复 → 重新验证
```

### 失败修复流程

1. 输出验证报告，列出所有「失败」检查项
2. 对每个失败项，说明具体问题和修复建议
3. Agent 修复问题
4. 重新运行完整验证清单（**全部重新检查，不跳过已通过项**）
5. 重复直到全部通过

### 升级机制

**连续 3 次验证失败**（同一任务修复后仍无法通过）→ 升级至用户：

```
## 验证升级报告

任务：[任务描述]
失败次数：3 次

未通过的检查项：
- [检查项编号] 描述
  - 第一次失败原因：...
  - 第二次失败原因：...
  - 第三次失败原因：...

影响评估：[说明未通过项对系统的影响]

建议：[手动处理建议]
```

## 验证报告格式

每次验证完成后，输出以下格式的报告：

```
## 验证报告

### 代码规范检查
- [通过] 1.1 Domain 反模式
- [通过] 1.2 命名规范
- [失败] 1.3 错误处理 → handler 中忽略了 ctx.Err()
- [通过] 1.4 金额规范
- [N/A] 1.5 状态机规范
- [N/A] 1.6 认证规范

### 测试检查
- [通过] 2.1 L1 单元测试通过
- [通过] 2.2 覆盖率达标（当前: 85%, 要求: >= 80%）
- [N/A] 2.3 L2 场景覆盖
- [通过] 2.4 测试数据无硬编码
- [通过] 2.5 测试反模式

### Git 检查
- [通过] 3.1 Commit message 格式
- [通过] 3.2 变更文件完整性
- [通过] 3.3 无敏感信息
- [通过] 3.4 单任务单提交

### 文档检查
- [通过] 4.1 OpenAPI spec 更新
- [N/A] 4.2 枚举注册
- [N/A] 4.3 错误码注册

## 结论：[验证通过 / 验证失败（1 项未通过）]
```

## 与其他 Skill 的协作

- **`storeforge-harness`**：验证清单 1.1 直接依赖 harness 的约束检查结果
- **`storeforge-testing`**：验证清单 2.1-2.5 按 `storeforge-testing` 定义的三层测试标准执行
- **`storeforge-executing`**：每个子任务完成后自动触发本验证
- **`storeforge-review`**：review 通过后的最终交付前，再次触发本验证
